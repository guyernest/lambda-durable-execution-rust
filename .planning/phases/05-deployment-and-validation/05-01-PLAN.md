      - AttributeName: agent_name

**New Resource: McpAgentFunction** — follows exact same pattern as existing functions, but with additional IAM policies for DynamoDB and Secrets Manager (per DEPL-01, DEPL-02):
```yaml
McpAgentFunction:
  Type: AWS::Serverless::Function
  Metadata:
    BuildMethod: rust-cargolambda
    BuildProperties:
      Binary: mcp_agent
  Properties:
    CodeUri: .
    Timeout: 900
    MemorySize: 256
    Policies:
      - AWSLambdaBasicExecutionRole
      - arn:aws:iam::aws:policy/service-role/AWSLambdaBasicDurableExecutionRolePolicy
      - Statement:
          - Effect: Allow
            Action:
              - lambda:CheckpointDurableExecution
            Resource: "*"
          - Effect: Allow
            Action:
              - dynamodb:GetItem
            Resource: !GetAtt AgentRegistryTable.Arn
          - Effect: Allow
            Action:
              - secretsmanager:GetSecretValue
            Resource: "*"
    Environment:
      Variables:
        AGENT_REGISTRY_TABLE: !Ref AgentRegistryTable
```

Key differences from other examples:
- `Timeout: 900` (15 min — agent may need many LLM round trips, overrides Globals 30s)
- `MemorySize: 256` (LLM response parsing needs more than 128MB default)
- Additional IAM: `dynamodb:GetItem` scoped to AgentRegistryTable ARN, `secretsmanager:GetSecretValue` for API key retrieval
- `AGENT_REGISTRY_TABLE` env var pointing to the DynamoDB table

Create `examples/scripts/validate_agent.py` — a standalone Python script (per D-02, D-03) that validates a deployed mcp_agent Lambda end-to-end.

**Script header** (PEP 723 inline metadata, same pattern as validate.py):
```python
# /// script
# requires-python = ">=3.11"
# dependencies = [
#     "click>=8.0",
#     "boto3>=1.35",
# ]
# ///
```

**CLI interface** (per D-03):
```
@click.command()
@click.option("--region", default="us-east-1")
@click.option("--stack", default="durable-rust")
@click.option("--mcp-server-url", required=True, help="URL of MCP server to validate against")
@click.option("--agent-name", default="test-agent")
@click.option("--version", default="v1")
@click.option("--timeout-seconds", default=240, type=int)
@click.option("--poll-seconds", default=3.0, type=float)
@click.option("--keep-config", is_flag=True, help="Don't delete AgentRegistry item after validation")
```

**Implementation steps:**

1. **get_stack_outputs(cfn_client, stack)** — same pattern as validate.py, returns dict of output keys to values. Need `McpAgentFunctionArn` and `AgentRegistryTableName`.

2. **seed_agent_config(dynamodb_client, table_name, agent_name, version, mcp_server_url)** — Put item into AgentRegistry with:
   - `agent_name` (S): from --agent-name
   - `version` (S): from --version
   - `system_prompt` (S): "You are a helpful assistant. Use the available tools to answer the user's question. Be concise."
   - `llm_provider` (S): "anthropic"
   - `llm_model` (S): "claude-sonnet-4-20250514"
   - `parameters` (S): JSON string of `{"max_iterations": 5, "temperature": 0.0, "max_tokens": 1024, "timeout_seconds": 120}`
   - `mcp_servers` (S): JSON string of `["<mcp_server_url>"]`
   Uses `dynamodb_client.put_item()`.

3. **invoke_agent(lambda_client, function_arn, agent_name, version, user_message)** — Invoke the Lambda asynchronously (InvocationType="Event") with an AgentRequest payload:
   ```json
   {
     "agent_name": "<agent_name>",
     "version": "<version>",
     "messages": [{"role": "user", "content": {"content": "<user_message>"}}]
   }
   ```
   The `content` field uses the `Text` variant format: `{"content": "..."}` (matching `MessageContent::Text` serde). Return the DurableExecutionArn.

4. **poll_execution(lambda_client, durable_arn, timeout, poll_interval)** — Same polling pattern as validate.py: call `get_durable_execution()` in a loop, check for terminal status (SUCCEEDED, FAILED, TIMED_OUT, STOPPED). Return final execution info.

5. **get_execution_result(lambda_client, durable_arn)** — After SUCCEEDED, call `get_durable_execution_history()` and extract the final result from the last `ExecutionSucceeded` event's output data.

6. **validate_response(result_json)** — Check the AgentResponse JSON structure:
   - Has top-level `message` key (from flattened LLMResponse)
   - Has top-level `metadata` key (from flattened LLMResponse)
   - `metadata.model_id` is a non-empty string
   - `metadata.stop_reason` is present
   - Has `agent_metadata` key (observability data)
   - `agent_metadata.iterations` >= 1
   - `agent_metadata.total_input_tokens` > 0
   - `agent_metadata.total_output_tokens` > 0
   - `agent_metadata.elapsed_ms` > 0
   Returns list of validation errors (empty = pass).

7. **cleanup_agent_config(dynamodb_client, table_name, agent_name, version)** — Delete the seeded item (unless --keep-config). Uses `dynamodb_client.delete_item()`.

8. **Main flow:**
   - Print header (styled, like validate.py)
   - Load stack outputs
   - Seed agent config
   - Invoke agent with message: "What tools do you have available? List them briefly."
     (This is a simple question that triggers tool discovery but may not require actual tool execution, keeping validation fast. The LLM should respond with a description of available tools.)
   - Poll for completion
   - If SUCCEEDED: extract result, validate response structure, print details
   - If FAILED/TIMED_OUT: print error details
   - Clean up agent config (always, even on failure — use try/finally)
   - Print summary (PASS/FAIL)
   - Exit 0 on pass, 1 on fail

**Styled output** — reuse the same styling approach from validate.py (BOX characters, SYM_OK/SYM_FAIL, click.style with colors). Keep it simpler since there's only one example to validate.

**User message selection:** Use "What tools do you have available? List them briefly." — this forces the agent to load config and discover tools (exercising CONF and MCP paths) but the LLM will likely answer directly without calling tools, keeping the validation fast (~1 iteration). If the MCP server has a simple tool (like Calculator), the LLM might call it. Either path validates the full pipeline.
